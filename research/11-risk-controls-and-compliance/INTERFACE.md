# Risk Controls Interface

Status: Phase 0 interface sketch only. No production code. No trading code.

## Purpose

The risk controls interface should sit between prediction output and any paper ledger or future broker adapter. It must reject unsafe intents before they become simulated orders. Pre-trade rejection is recommended because SEC Rule 15c3-5 requires controls designed to prevent orders that exceed thresholds or appear erroneous: https://ecfr.io/Title-17/Section-240.15c3-5

## Inputs

- `DecisionContext`: decision timestamp, symbol, session state, market calendar state, halt state, LULD state, account mode, operator ID, and run ID. Include halt and LULD fields because LULD and market-wide circuit breakers can pause trading or prevent trades outside specified price bands: https://www.finra.org/filing-reporting/trf/limit-uplimit-down-luld-plan and https://www.investor.gov/introduction-investing/general-resources/news-alerts/alerts-bulletins/measures
- `MarketSnapshot`: NBBO, last trade, bar, volume, spread, top-of-book depth, data timestamps, data source, and freshness flags. Reject stale or missing NBBO because erroneous-order controls need price checks and FINRA warns system failures can affect execution: https://ecfr.io/Title-17/Section-240.15c3-5 and https://www.finra.org/rules-guidance/rulebooks/finra-rules/2270
- `FeatureSnapshot`: feature values, feature timestamps, source data cutoffs, and no-future-data proof. This is required because repository rules prohibit future data and the TRIDENT model uses timestamped data streams: ../../AGENTS.md and ../../docs/TRIDENT_LOB_MODEL.md
- `ModelSignal`: model ID, model version, calibration version, prediction horizon, side, score, confidence, expected holding period, and diagnostic flags. Reject invalid or stale model metadata because SEC Rule 15c3-5 requires controls against erroneous orders: https://ecfr.io/Title-17/Section-240.15c3-5
- `PortfolioState`: paper cash, paper equity, open paper positions, pending paper intents, realized PnL, unrealized PnL, gross exposure, net exposure, per-symbol exposure, margin mode, and day-trade counters. Include margin and day-trade fields because Regulation T, FINRA Rule 4210, and the pending intraday margin transition can affect account constraints: https://www.federalreserve.gov/frrs/regulations/section-22012-supplement-margin-requirements.htm, https://www.finra.org/rules-guidance/rulebooks/finra-rules/4210, and https://www.sec.gov/rules-regulations/self-regulatory-organization-rulemaking/sr-finra-2025-017
- `RiskConfig`: max daily loss, max position notional, max symbol exposure, max gross exposure, max orders per minute, price collar, allowed symbols, allowed sessions, short-sale permission, margin permission, and manual approval IDs. These fields are recommended because SEC Rule 15c3-5 calls for pre-set credit, capital, price, size, and access controls: https://ecfr.io/Title-17/Section-240.15c3-5
- `BrokerConstraints`: broker API rate limit, buying-power calculation, paper-vs-live mode, order-type availability, extended-hours availability, shortability, borrow fee availability, minimum order rules, and paper fill caveats. If Alpaca is used later, these fields should reflect Alpaca order, paper, and API rate-limit docs: https://docs.alpaca.markets/docs/trading/orders/, https://docs.alpaca.markets/docs/trading/paper-trading/, and https://alpaca.markets/support/usage-limit-api-calls

## Outputs

- `RiskDecision`: `APPROVE_PAPER_INTENT`, `REJECT`, `HALT_NEW_EXPOSURE`, or `MANUAL_REVIEW_REQUIRED`. The interface should never output `APPROVE_LIVE_ORDER` in Phase 0 or Phase 1 because the project scope blocks live trading: ../../AGENTS.md and ../../docs/TRIDENT_LOB_MODEL.md
- `RejectReason`: structured reason codes for daily loss, position size, symbol exposure, gross exposure, order rate, price collar, stale data, future data, model failure, halt state, LULD state, margin blocked, short blocked, broker constraint, duplicate intent, or missing approval. Structured rejection is recommended for documented controls under SEC Rule 15c3-5: https://ecfr.io/Title-17/Section-240.15c3-5
- `ApprovedPaperIntent`: symbol, side, quantity, order type for paper simulation, limit price if any, time in force, decision timestamp, expiration timestamp, risk config version, model version, and ledger ID. Include complete order lifecycle fields because broker APIs expose order states and paper fills can differ from live fills: https://docs.alpaca.markets/docs/trading/orders/ and https://docs.alpaca.markets/docs/trading/paper-trading/
- `AuditEvent`: append-only record containing input hashes, source timestamps, risk rule versions, decision, reject reason, operator ID, approval ID, and kill-switch state. Append-only audit logging is recommended by the documentation and post-trade report principles in SEC Rule 15c3-5: https://ecfr.io/Title-17/Section-240.15c3-5

## Required Rule Checks

- Mode check: reject unless mode is `DRY_RUN` or approved broker paper mode. Live mode must always reject in Phase 0 and Phase 1 under project rules: ../../AGENTS.md and ../../docs/TRIDENT_LOB_MODEL.md
- Account check: reject margin use unless margin simulation is explicitly approved and Regulation T, FINRA Rule 4210, selected broker house rules, and intraday margin rules are modeled: https://www.federalreserve.gov/frrs/regulations/section-22012-supplement-margin-requirements.htm, https://www.finra.org/rules-guidance/rulebooks/finra-rules/4210, and https://www.sec.gov/rules-regulations/self-regulatory-organization-rulemaking/sr-finra-2025-017
- Day-trade check: maintain both legacy PDT counters and broker-confirmed intraday margin fields until the selected broker confirms effective treatment. This is recommended because SEC approved SR-FINRA-2025-017, while FINRA public guidance still includes legacy PDT language: https://www.sec.gov/rules-regulations/self-regulatory-organization-rulemaking/sr-finra-2025-017 and https://www.finra.org/rules-guidance/guidance/interps-4210
- Short-sale check: reject short simulation unless the symbol is shortable, locate status is present, and close-out risk is modeled. This follows Regulation SHO locate and close-out requirements: https://www.sec.gov/investor/pubs/regsho.htm and https://ecfr.io/Title-17/Part-242/SubjectGroup-regulation-sho-regulation-of-short-sales
- Data freshness check: reject stale, out-of-order, missing, or future-timestamped data. This follows repository no-future-data rules and mitigates system-failure risks described by FINRA: ../../AGENTS.md and https://www.finra.org/rules-guidance/rulebooks/finra-rules/2270
- Model health check: reject invalid model output, stale model version, schema mismatch, or missing calibration. This supports SEC Rule 15c3-5 erroneous-order controls: https://ecfr.io/Title-17/Section-240.15c3-5
- Exposure check: reject if the intent breaches daily loss, single position, symbol exposure, gross exposure, or concentration limits. This implements pre-set credit and capital thresholds under SEC Rule 15c3-5: https://ecfr.io/Title-17/Section-240.15c3-5
- Order-rate check: reject when global or per-symbol order rates exceed config, and keep caps below broker API throttles. For Alpaca paper or trading APIs, use 200 requests per minute as a hard broker throttle unless broker documentation changes: https://alpaca.markets/support/usage-limit-api-calls
- Price check: reject marketable or nonmarketable paper intents outside approved collars, during LULD states, or when NBBO is unavailable. This follows SEC erroneous-order controls and LULD price-band principles: https://ecfr.io/Title-17/Section-240.15c3-5 and https://www.finra.org/filing-reporting/trf/limit-uplimit-down-luld-plan
- Manual approval check: reject restricted operations without named approval. Restricted operations include paper broker enablement, short simulation, margin simulation, risk-limit changes, model promotion, data vendor changes, and live-readiness review. This follows FINRA account approval concepts and SEC access-control principles: https://www.finra.org/rules-guidance/rulebooks/finra-rules/2130 and https://ecfr.io/Title-17/Section-240.15c3-5

## Logging Contract

Every risk decision should produce exactly one audit event. Every kill-switch transition should produce a separate audit event. Every manual approval should include approver, timestamp, scope, expiry, and linked checklist. This logging is recommended to support documented controls and immediate post-trade reporting concepts under SEC Rule 15c3-5: https://ecfr.io/Title-17/Section-240.15c3-5
